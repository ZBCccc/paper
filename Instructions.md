# 论文系统方法论（学术写作版）

## 目的

本文件用于把原始 `Instructions.md` 的系统化方法论，迁移为你的学术论文工作流。
目标是让 AI 在同一工作目录中持续复用你的研究上下文，而不是每次从零开始。

## 你的当前场景

- 研究者：张柏川（华中科技大学网络空间安全硕士）
- 研究方向：密码学、可搜索加密（SSE）
- 当前主工程：`HUST-PhD-Thesis-Latex`
- 当前论文主题：多用户多关键词 SSE 的可验证机制（资格检验 + 结果正确性）

## 三条设计原则（迁移版）

### 1. 按研究流程分类，不按文件类型分类

论文写作时真正关心的是“研究做到哪一步”，而不是“这个文件是 PDF 还是 MD”。
因此目录按流程组织：选题管理、素材库、方法论、数据统计。

### 2. 共享层 + 论文业务层

- 共享层：`CLAUDE.md`、`个人定位.md`、`记忆库/`、`04-技能库/`
- 论文业务层：`01-论文生产/` + `HUST-PhD-Thesis-Latex/`

共享层用于长期记忆与协作规范；业务层用于具体产出与迭代。

### 3. 生命周期管理

选题生命周期：

1. `00-选题记录.md`：收集所有想法
2. `01-待深化/`：可行但证据不足
3. `02-待投稿/`：主体完成、待打磨
4. `03-已投稿/`：已投稿与归档

章节生命周期（针对当前论文）：

1. 草稿完成
2. 理论证明补齐
3. 实验验证补齐
4. 终稿润色

## 日常执行规范

## 论文语言硬约束（全局）

1. 禁止在论文正文中使用“写作说明式元叙述”，例如“为符合可证明安全论文的写作口径”“为便于叙述”“下文将介绍本文结构”等。
2. 仅保留与模型、定义、假设、定理、证明、实验结论直接相关的客观陈述。
3. 安全章节优先使用标准形式化表达：PPT 对手、泄露函数、Real/Ideal（或 Game）定义、可忽略函数界。
4. 改写与润色时，若出现“作者在写作/组织内容”的句子，应先删除再重写为技术陈述。
5. 默认协作角色为“密码学研究者 + 论文写作教练”，输出应达到可直接并入论文正文的学术质量。
6. 禁止模板化 AI 句式（如“本文将/我们将/总之/显而易见/值得注意的是/随着……发展”）与空泛形容词（如“巨大”“革命性”“非常重要”）。
7. 不得编造文献、定理、参数、实验结果；缺失信息统一标记为【待补充】。
8. 安全证明遵循“游戏/混合实验 + 归约”规范结构，明确对手模型、优势函数、归约算法、失败事件与界推导。
9. 协议章节一律使用密码学标准术语（初始化、更新、令牌生成、检索、验证），避免“写路径/查路径”等非标准口径。
10. 对“基于已有方案改进”的章节，默认结构为：基线能力与边界 -> 可验证性缺口与攻击面 -> 改造目标与技术路线。
11. 涉及 RBF/BF 时，必须显式说明参数语义与协议映射关系（例如 $\ell,k,\beta_i$ 与 Update/GenToken/Search 的对应），并交代其与可验证目标的张力。

### 每日（10-20 分钟）

1. 把当日新增想法写入 `01-论文生产/选题管理/00-选题记录.md`
2. 同步章节进度到 `控制面板.md`
3. 若章节结构发生变化，更新 `01-论文生产/方法论/论文研究流程.md`
4. 新增 PDF 文献时，按 `ljg-xray-paper` 抽取为 Markdown 并存入 `01-论文生产/素材库/论文Markdown库/`

### 每周

1. 清点 `HUST-PhD-Thesis-Latex/body/chapter/` 的 TODO
2. 补齐一个“最卡脖子”的模块（证明/实验/图表三选一）
3. 将本周关键结论沉淀到 `记忆库/`

### 每月

1. 评估目录是否仍符合你的“第一反应”
2. 清理长期未使用模板，避免系统臃肿
3. 回顾投稿路径并更新 `01-论文生产/数据统计/投稿记录.md`

## 系统健康检查（沿用原方法）

每次迭代后检查三件事：

1. 找资料时，第一反应路径是否正确？
2. 是否出现长期不使用但持续维护的目录？
3. 新内容入库时是否犹豫放哪？

若有任一“是”，说明结构需要继续优化。

## 初始化完成标准

若满足以下条件，说明论文系统已可长期使用：

1. 论文核心问题、创新点、章节状态已在 `记忆库/` 固化
2. 选题与任务已进入生命周期轨道（记录/待深化/待投稿/已投稿）
3. `HUST-PhD-Thesis-Latex` 与方法论文件形成双向映射（源码改动可回写到系统）
