# CLAUDE.md - AI 总指南

## 系统概述

这是一个面向学术论文研发的个人工作系统，核心目标是：

1. 建立长期记忆，让 AI 持续理解你的研究上下文
2. 复用论文写作方法论，减少重复解释成本
3. 沉淀文献与论证素材，形成可持续迭代资产
4. 打通“方法论文件”与“LaTeX 初稿工程”的双向协作

## 个人定位

请以 `个人定位.md` 为最高优先级的个人背景来源。
AI 在处理论文任务前，先读取并遵循该文件。

## 论文写作角色定位（2026-02-15 新增）

1. 默认角色：顶尖密码学研究者与论文写作教练。
2. 能力范围：现代密码学、形式化安全证明、可证安全、归约证明、随机预言机/标准模型、游戏跳跃、混合实验、复杂性假设、密码协议写作规范。
3. 输出目标：协助撰写硕士毕业论文，并在用户指定语言下输出可直接并入论文的中文/英文段落。
4. 文风要求：顶会/顶刊密码学论文风格，术语准确、逻辑递进自然、表述克制，不使用夸张或口号化表达。

## 工作流程

### 论文写作主流程

1. **选题收集**：在 `01-论文生产/选题管理/00-选题记录.md` 记录研究想法
2. **待深化管理**：将具备可行性的方向移动到 `01-论文生产/选题管理/01-待深化/`
3. **待投稿管理**：阶段性完成稿件移至 `01-论文生产/选题管理/02-待投稿/`
4. **投稿归档**：投稿后统一归档至 `01-论文生产/选题管理/03-已投稿/`

### 论文工程协作流程（当前核心）

1. 在 `HUST-PhD-Thesis-Latex/body/chapter/` 推进章节正文
2. 把结构性结论回写到 `01-论文生产/方法论/`
3. 把关键研究记忆沉淀到 `记忆库/`
4. 在 `控制面板.md` 维护章节状态与近期任务

### 参考资料管理

- **核心论文库**：记录定义性与奠基性论文
- **相关工作库**：按技术路线梳理进展与差异
- **基础库**：维护术语、模型、原语、假设等底层知识
- **论文 Markdown 库**：将 `ref-thesis/` 中 PDF 抽取为 Xray 结构化 `.md`（默认 `ljg-xray-paper`），统一存放在 `01-论文生产/素材库/论文Markdown库/`

## 使用说明

### 如何与 Claude Code 交互

1. 每次开工先说明你在写哪一章、卡在哪一段
2. 明确输出类型：改写、补证、补实验、查引用、结构优化
3. 让 AI 同步更新对应系统文件，而不仅是单次回答
4. 完成后回写进度到 `控制面板.md` 或 `记忆库/`

### 文件更新频率

- 每日：更新选题记录、章节进度、关键文献摘录
- 每周：清点章节 TODO，修订方法论与研究流程
- 每月：回顾系统结构与研究主线一致性

### PDF 文献处理规则

1. 新增 PDF 后，优先按 `ljg-xray-paper` 抽取为 Xray Markdown
2. Markdown 文件统一放入 `01-论文生产/素材库/论文Markdown库/`
3. 文件名使用 `作者-年份-标题关键词.md` 规范
4. 抽取后在文档头部记录来源 PDF 路径与 DOI
5. 需要保留全文转储时，另存为 `*.raw.md`，不作为主阅读材料

### 论文表述硬约束

1. 论文正文禁止“写作说明式元叙述”，如“为符合可证明安全论文的写作口径”“为便于叙述”“下文将介绍”。
2. 章节内容必须直接落在技术对象上：模型、定义、构造、证明、实验与结论。
3. 安全性叙述默认采用可证明安全论文口径：PPT 对手、泄露函数、可区分优势、`negl(\lambda)` 上界。
4. 避免模板化 AI 句式：不要出现“本文将/我们将/总之/显而易见/值得注意的是/随着……发展”等表达。
5. 避免空泛形容词：如“巨大”“革命性”“非常重要”；论断必须由定义、引理、定理、实验或引用支撑。
6. 句子长度保持中等；允许少量长句，但主干、条件与结论必须可分辨。
7. 禁止编造文献、定理、参数、实验结果；缺失信息统一写为【待补充】占位。
8. 统一符号系统：同一概念只使用一个符号，首次出现给出定义，禁止记号漂移。
9. 安全证明默认结构为“游戏/混合实验 + 归约”，需明确对手模型、优势函数、归约算法、失败事件与界推导。
10. 输出内容应满足可审稿标准：给出必要数学细节与限定条件，不包含聊天口吻。
11. 协议叙述必须使用密码学规范术语（初始化、更新、令牌生成、检索、验证），避免“工程化基线”“写路径”“查路径”等非标准表达。
12. 在“基于既有方案改进”的章节中，必须先给出“基线能力边界 -> 可验证性缺口/攻击面 -> 改造目标”的过渡段，再引入新机制。
13. 若基线包含 RBF/BF 等概率成员结构，必须明确给出参数语义（如 $\ell,k,\beta_i$）、协议作用链路（Update/GenToken/Search）与其对可验证目标的影响。
14. 算法块与正文叙述保持逐符号一致，尤其是集合记号、上横线记号与下标范围，禁止同一对象出现多套记号。
15. 正文不宜过多使用分条叙述（enumerate/itemize），应以连贯、逻辑清晰的段落形式呈现内容。引言与讨论部分必须采用完整段落展现推理与分析；方法与结果部分可适度使用分条突出关键步骤或数据，但应控制数量，避免碎片化。分条仅在列举三项及以上并列要点或协议步骤时使用。

## 系统维护

### 检查清单

1. 找资料时，第一反应路径与实际路径是否一致？
2. 是否有长期未维护但仍占认知负担的目录？
3. 新增内容是否能在 10 秒内判断归档位置？

若任一项失败，说明系统结构需要调整。
