## 三、关键写作经验（可复用）

### 安全证明写作规范

1. **归约算法必须说明模拟能力**：每个 $\mathcal{B}_i$ 必须明确：(a) 持有哪些密钥，(b) 缺少哪个密钥（由挑战者提供预言机），(c) 如何模拟实验的全部交互。仅写"构造与命题 X 相同"不够，至少需一句话说明密钥持有与模拟方式。

2. **条件性质需显式声明前提**：当安全性质依赖其他性质保证的前提时（如 VQ-Sound 依赖地址归属正确），必须在实验定义中写明前提条件，并在命题声明中标注"条件性质"。

3. **认证加密 vs 对称加密**：当载荷需要完整性保护时，必须使用认证加密（AE）而非普通对称加密（SE），对应的安全归约目标是 INT-CTXT 而非 CPA/PRF。

4. **组合安全实验必须自包含**：组合实验（如 CV）不能仅引用子实验，必须完整写出预言机定义、状态维护和验证流程，因为组合实验的验证流程通常比子实验更复杂。

5. **对比表必须覆盖泄露函数维度**：VSSE 方案对比不能只比较验证目标和开销，泄露函数是区分方案安全性的关键维度。

6. **交叉引用的归约算法仍需独立说明**：即使归约构造与前文命题相同，在新命题中仍需简要说明模拟能力，因为读者可能不会回翻。

### 工作流经验

7. **问题驱动的批量修复**：先列出所有问题（问题列表），再按依赖关系排序修复，最后统一编译验证。避免逐个发现逐个修的碎片化模式。

8. **Markdown-LaTeX 双向同步**：方法论 Markdown 中的修改必须同步到 LaTeX，反之亦然。本轮采用"先修 LaTeX 再同步 Markdown"的顺序，效率更高。

9. **子模块提交顺序**：先在子模块内 commit+push，再在主仓库 commit+push，否则主仓库会记录 dirty submodule。
